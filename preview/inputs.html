Design system

Inputs

Form controls — the entry points for user data. Each control captures a single piece of information; together they compose forms, search bars, settings panes, and onboarding flows. Built around a shared .input base with consistent sizing, focus, validation, and inverse rules.

01Anatomy

Every field is a vertical stack: label, field row, helper or error. The field row holds the input plus optional affixes (icons, prefix/suffix, trailing buttons). Spacing between rows is --space-2; between fields, --space-5.

02Variants

Twenty variants, organised in five families: text fields (the .input base + affixes), structured fields (group, select, number-input, date), specialised fields (chips, OTP, file, slider), boolean (switch, checkbox, radio), and quiet (underline, pill search). All consume the same focus, error, disabled tokens.

Three sizes — pick by surface density. Default (40px) for most product surfaces. Small (32px) for dense data tables and toolbars. Large (48px) for hero forms (sign-up, onboarding) and mobile-first contexts where touch targets must be generous.

04States matrix

Five canonical states for every text-flavour field: default, :hover (border darkens), :focus (border becomes brand blue + 4px focus ring), filled (value present, neutral border), and validation .is-error. Disabled is universal — opacity 0.5, no interaction. Selection & boolean controls collapse this to off / on / focus / disabled.

05Composition

How fields combine into real forms. The atomic unit is the field — a label, the input itself, an optional helper or error message. Fields stack vertically inside a fieldset; fieldsets stack inside a form. Use grid for multi-column layouts (1 column on mobile, 2–3 on desktop), but never split a single logical field across columns.

Field anatomy

The smallest composable unit. <label> always sits above the input; helper or error text sits below. Required marker is a non-decorative asterisk in --color-feedback-danger-foreground.

06On inverse

Inputs on dark surfaces — sign-in screens, marketing forms, dashboards in dark mode. Border, background, and label colours flip to inverse tokens; focus colour stays brand blue (it has enough lightness to read on both). Validation colours don't change — error red and success green are tuned for both surfaces in the colour system.

07Don'ts

Patterns that look reasonable in isolation but break the system. Each example pairs the wrong with the right.

Don't · placeholder as label

Placeholder disappears on type. Users forget what the field was for, and screen readers don't always announce it.

Wrong

Do · always label above

Label persists. Placeholder shows format hint.

Right

Don't · red borders without text

Colour alone fails for users with low colour vision. Error must be paired with a message.

Wrong

Do · pair colour with text

Specific, actionable error message. Icon reinforces the state non-redundantly.

Right
Email must include a domain.

Don't · split a field across columns

The label belongs to the input. Splitting them across columns confuses scanning and breaks responsive flow.

Wrong

Do · stack label above input

Vertical pair — scans top-to-bottom, works at any width, accessible to AT.

Right

Don't · disabled looks like filled

If a disabled input is too similar to a filled one, users try to edit it and get frustrated.

Wrong

Do · disabled has reduced opacity

0.5 opacity + not-allowed cursor + --color-background-subtle read clearly as inactive.

Right

Don't · use Switch for forms

Switches imply immediate effect. In a form (where the user must press Save), they mislead the user into thinking the change has applied.

Wrong

Do · use Checkbox for forms

Checkbox state is captured on submit — matches the mental model of "I'll review and confirm".

Right

Don't · radio for boolean

Two radio buttons for yes/no waste space and read as a setting in flux.

Wrong
Email me updates?

Do · single checkbox

One checkbox = one boolean. The label asks the question.

Right

08Accessibility

Inputs are the most-touched control type in any product. They must work flawlessly with keyboard, screen readers, voice control, and touch. The rules below are non-negotiable — every variant in this document already complies.

Requirements

Mapped to WCAG 2.2 AA. Each rule has a concrete implementation.

RequirementHow we meet itWCAG
Visible labelEvery input has a <label for="…"> above it. Placeholder is supplemental, never a replacement.3.3.2
Associated labelEither wrap the input or use for+id. Boolean controls wrap (<label><input>…</label>).1.3.1, 4.1.2
Visible focus4px --shadow-focus ring on every focusable element. Never outline: none without an alternative.2.4.7, 2.4.11
Touch target≥ 32px on all sides for default size. Boolean controls have a 32px clickable label even when the visual is smaller.2.5.8
Colour + text for stateError state always pairs --color-danger-500 border with a text message. Success likewise. Never colour-only.1.4.1
ContrastText 4.5:1 minimum. Borders 3:1 against background. Tested on light & inverse surfaces.1.4.3, 1.4.11
Error associationHelper / error message has id; input has aria-describedby pointing to it. Errors also set aria-invalid="true".3.3.1, 4.1.3
Required markerVisual asterisk in --color-feedback-danger-foreground + aria-required="true". Never asterisk-only.3.3.2
Autocompleteautocomplete hint on personal-data fields (email, name, postal-code, cc-number…) so password managers and AT can pre-fill.1.3.5
Disabled vs read-onlydisabled when the field is non-applicable. readonly when the value matters but can't be changed (still focusable, still in tab order).1.3.1
Group namingRadio & checkbox groups always wrap in <fieldset> with a <legend>. Never a free-floating heading.1.3.1
Switch roleSwitch markup is <input type="checkbox" role="switch">. AT announces "switch, on/off" rather than "checked/unchecked".4.1.2
Mobile keyboardsUse type="email", type="tel", inputmode="numeric", inputmode="decimal" for numeric and currency fields.1.3.5
MotionAll transitions ≤ 160ms, no spinning or strobing. Honors prefers-reduced-motion.2.3.3
