Design system

Tooltip

A small, transient label that names or briefly explains a target. Triggered by hover or keyboard focus, never carries critical information, never holds interactive content.

01Anatomy

A tooltip has two parts: the target it names, and the floating label itself with a small caret pointing back to the target.

Settings
  1. TargetThe element being labeled. Must be focusable for keyboard users — typically a button, icon, or chip.
  2. Label surfaceInverse-tinted, low-elevation pill. The visual contrast separates it from regular UI text.
  3. Caret3px arrow pointing at the target. Connects label to source visually.
  4. Offset8px gap between target and tooltip. Keeps the caret readable at small sizes.

03Variants

The default. Short, single-line label. Use for icon buttons and any control whose meaning isn't obvious without text.

04Triggers & timing

Show on hover after a small delay; show immediately on keyboard focus. Hide on leave, on Escape, and when the target loses focus.

EventBehaviorTiming
Mouse enterShow with a smooth fade + slideNo delay — --duration-base (200ms) --ease-emphasis
Mouse leaveHideImmediate
Keyboard focusShow immediately0ms — keyboard users want the label now
Keyboard blurHideImmediate
EscapeHide while focus stays on targetImmediate
Touch tapShow, then hide on next tap or scrollLong-press only — single tap activates the underlying control
Multiple targets in a rowIf a tooltip is already open within ~300ms, others show without delay"Warm-up" pattern — once a user is inspecting, follow-ups are instant

05Use it when…

The decision tree: tooltip vs. popover vs. label.

NeedUseWhy
Naming an icon-only buttonTooltipShort, on-demand, non-essential to read first.
Showing a keyboard shortcutTooltip (with kbd)Surfaces the shortcut without taking layout space.
Defining a financial termTooltip richOne-sentence definition, no interaction needed.
Long explanation, multiple paragraphsPopover insteadTooltips can't be tapped to keep open. Popovers can.
Information user MUST seeInline help text insteadTooltips are missed by screen-only scanners. Hover ≠ visibility.
Confirming a successful actionToast insteadTooltips don't appear without intent.
Linking to another pagePopover or just a regular linkTooltips can't hold interactive content.

06Don'ts

Tooltips are easy to misuse. Each of these patterns degrades accessibility or breaks the contract that tooltips are quick, optional reminders.

Interactive content insideTooltips disappear on mouse leave. Links and buttons inside are unreachable for keyboard users. Use a Popover.

Save changes

Repeating the visible labelIf the button has a text label, the tooltip is noise. Reserve them for icon-only or genuinely cryptic targets.

Delete (disabled)

Tooltip on a disabled button[disabled] elements don't fire mouse events in most browsers — and aren't focusable. Wrap the button in the target span, or replace disabled with aria-disabled.

07Accessibility

ConcernRequirementHow it's met
Keyboard accessibilityTooltip shows on focus, hides on blur and EscapeTarget is focusable; tooltip uses :focus-visible and :focus-within selectors. Escape handler in JS.
RoleWAI-ARIA tooltipLabel has role="tooltip" and unique id; target references it via aria-describedby.
Hover persistenceWCAG 1.4.13 — content is dismissable, hoverable, persistentCursor can move from target onto tooltip without it disappearing (target span hosts both). Esc dismisses without removing focus.
TouchNo content reachable only by hoverAll tooltip information has a text alternative (button label, inline help, or aria-label).
Color contrastLabel vs. surface ≥ 4.5:1 (AA)Inverse foreground (--color-foreground-inverse-default) on inverse background (--color-background-inverse) — passes AAA.
Reduced motionRespects prefers-reduced-motionSlide-in is removed; only a fast opacity fade remains.
Disabled targetsTooltip still announceableWrap the disabled control in a span.tooltip-target with tabindex="0" so the wrapper carries focus and aria-describedby.
