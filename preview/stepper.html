Design system

Stepper

Multi-step flow indicator. Horizontal for wizards at the top of a form; vertical for long onboardings in a sidebar. Four states per step — todo, current, done, error — plus optional skipped. Not to be confused with Number Input, which is the numeric field with ± buttons.

01Anatomy

Three parts per step — marker, label, connector — wired together by the parent .stepper container. Each step is a list item; semantic ordering belongs in markup, not just visuals.

  1. Counterparty
  2. 2
    AmountSet total & currency
  3. 3
    Review
  1. Container .stepperSemantic <ol>. Default is horizontal; add .is-vertical to stack. Always set aria-label describing the flow ("Account opening", "Wire transfer").
  2. Step .stepOne <li> per stage. Carries the state modifier — .is-done, .is-current, .is-error, .is-skipped.
  3. Marker .step-markerThe circle. Number for todo / current, checkmark icon for done, alert icon for error. Use <a> if the step is jumpable, <span> if not.
  4. Body .step-bodyHolds .step-label (always) and optional .step-help (one short line, vertical orientation only).
  5. ConnectorDrawn automatically with ::after on every step except the last. Tints to the success token between done steps.

02Orientations

Pick the orientation by where the stepper lives, not by step count. Top-of-form ≤ 5 steps → horizontal. Sidebar & long onboardings ≥ 4 steps with helper copy → vertical.

Horizontal .stepper

Default. Lives above the form. Labels sit under markers, capped at ~16ch for compact alignment. Use this for transactional wizards (wire transfer, KYC quick-check).

03States

Exactly one step at a time is current. Multiple steps can be done; multiple can be error. Use skipped sparingly — only for genuinely optional steps the user deliberately bypassed.

05When to use

Steppers are for sequential, finite, known flows where the user benefits from seeing where they are and what's left. They're not progress bars, not tabs, not breadcrumbs.

SituationUseWhy
3–6 step wizard at the top of a formStepper · horizontalCompact, scannable, sets expectation up front.
Long onboarding with side panelStepper · verticalRoom for helper text per step; mirrors document outline.
Live status of an async process (transfer settling)Status pills or timeline, not stepperStepper implies user agency; status is observed, not driven.
Browsing between siblings (Overview, Members, Settings)TabsTabs are non-sequential.
Page hierarchy / context (Org > Project > Task)BreadcrumbsPosition in a tree, not sequence.
Single page-load progress (percent complete)Progress barContinuous, not discrete.
> 7 stepsRe-think the flow firstIf you can't get under 7, the form is likely two flows pretending to be one.

08Accessibility

ConcernRequirementHow it's met
SemanticsOrdered, navigable listRoot is <ol> with aria-label. Each step is <li>. AT announces "list, N items".
Current stepProgrammatically exposedThe current step's marker (or its wrapper) carries aria-current="step".
Visited vs upcomingPast steps are real links; future aren'tDone steps use <a> with real href. Todo steps use <span> so they're skipped in tab order.
Icon meaningNot by colour aloneDone = checkmark icon. Error = triangle-alert icon. Both also carry a colour shift, but the icon is the primary cue.
Error contextWhy & how to recoverError step uses .step-help with the recovery action. Pair with the form's inline error message.
Focus orderMatch visual orderMarkup order is also tab order. Don't reorder visually with CSS order.
Color contrastBorders ≥ 3:1, text ≥ 4.5:1All token-driven. Verified: muted on canvas = 4.6:1, success-fg on white = 4.8:1, danger-fg on white = 5.1:1.
Reduced motionNo transitions when requestedprefers-reduced-motion kills marker transitions.
