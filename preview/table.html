Design system

Table

Dense tabular-numeric data — the workhorse of the dashboard. A .dt-wrap card stacks an optional toolbar, the .dt table itself, and a pagination footer. Three sizes, opt-in zebra / sticky / bordered density, and a full set of cell patterns and row states. Numbers align by default via tabular-nums.

07Pagination

The footer that anchors a paged table. It answers two questions at a glance — where am I (.range) and how do I move (.pager). Lives at the bottom of .dt-wrap, mirroring the toolbar at the top.

Anatomy .dt-pagination

A left-aligned range readout and a right-aligned .pager cluster. The current page is .page-btn.is-current; the ends disable when there's nowhere to go.

Footer
1–6 of 248
  1. Range .range — mono “X–Y of N”. Always show the total so users gauge depth before paging.
  2. Pager .pager — right-aligned cluster of .page-btn; pushes to the edge via margin-left:auto.
  3. Prev / Next — chevron .page-btn; set disabled on the first/last page (greys to 40%).
  4. Current .is-current — the active page, tinted brand-secondary. Exactly one at a time.
  5. Truncation “…” — a non-interactive .page-btn standing in for skipped page ranges on long sets.
  6. Rows per page — optional control on the left, paired with the range (see below).

DecisionGuidance
Numbered pages vs Prev/Next onlyUse numbered pages when users jump around (search results, ledgers). Prev/Next-only suffices for linear review queues.
When to truncate with “…”Past ~7 page buttons. Always keep first, last, current and current±1 reachable; collapse the rest.
Pagination vs infinite scrollPaginate for data users act on, audit or cite a position in (page 3 of results). Infinite scroll only for exploratory feeds — never for financial records.
“Load more” buttonA middle ground for mobile / append-style lists. Don't mix it with numbered pages in the same view.
Empty & single-pageHide the pager when there's one page; keep the range readout (“6 of 6”) for consistency.

10Responsive & mobile

A grid of columns can't survive a 390px screen intact. Pick one of two strategies per table — never let columns silently crush.

StrategyHowUse when
A · ScrollWrap .dt in an overflow-x:auto element. The grid stays whole and pans sideways.Comparison-heavy / numeric tables where column alignment is the point — ledgers, FX rates, reconciliation.
B · Stack .dt-responsiveBelow 560px the wrapper collapses each row into a label : value card. Driven by a container query, so it also reflows inside drawers & split-panes.Action-oriented record lists the user taps into — transactions, counterparties, requests.

Stacked cards .dt-responsive

The same transactions table inside a phone frame. Every <td> carries a data-label that becomes the inline field label; the lead cell gets .dt-card-title to read as the card heading.

Strategy B

What changes at <560px

The container query (not the viewport) drives the switch — resize a desktop window or open this table in a drawer and it reflows the same way.

  • thead is hidden; column names move inline via data-label.
  • Each tr becomes a bordered card with the lead cell as its title.
  • Numbers stay right-aligned opposite their label for fast scanning.
  • Status pills, deltas and IDs keep their exact tokens — no restyle.

11When to use

Tables are for comparing many records across the same set of attributes. If users scan down a single column to compare values, a table is right. If each record needs rich, varied layout, use Cards instead.

Right-align all numbersUse .dt-num so decimals line up. Tabular figures are on by default at the .dt level.

Don't center-align numeric columnsCentered figures break vertical scanning and make magnitude impossible to compare.

Keep one primary action per rowA trailing .dt-actions cluster or a clickable row — not a wall of buttons in every cell.

Don't exceed ~7 columns on desktopPast that, move secondary attributes into a .dt-cell-stack sub-line or a detail drawer.

Always show count + paginationUsers need to know the total and where they are. Pair .count .total with .dt-pagination .range.

Don't tint rows per categoryRow colour means state (selected / error / muted) only. Use a status pill or lead tone for category.

12Accessibility

ConcernRequirementHow it's met
SemanticsReal table elementsNative <table><thead><tbody><th><td> — never <div> grids. Screen readers announce row/column position.
Header scopeColumn headers associatedAdd scope="col" on <th> (and scope="row" for row headers) in production markup.
Sort stateAnnounced, not colour-onlySortable header gets aria-sort="ascending|descending"; the chevron rotation mirrors it visually.
SelectionCheckboxes are real inputsUse a real <input type="checkbox"> with a label (the demo uses styled stand-ins). Bulk toolbar appears with aria-live="polite".
Row actionsReachable by keyboardIcon buttons carry aria-label; clickable rows expose the same action via a focusable control, not just the row click.
Empty / loadingCommunicated to ATSkeleton rows are aria-hidden with an aria-busy="true" table; empty state uses real text, not just an icon.
ContrastText ≥ 4.5:1, borders ≥ 3:1All token-driven. Zebra tint stays above 3:1 against body text; status pills carry text labels, not colour alone.
