Design system

Tabs

Non-sequential navigation between sibling views that share one context. Two flavours: .tabs (underline) for primary panel navigation, .tabs-pill (segmented) for compact filters and toolbars. Count badges, icons and full keyboard support. For sequential flows use the Stepper instead.

01Anatomy

A .tabs container holds a row of .tab buttons. The selected tab carries aria-selected="true", which drives both the brand-coloured label and the 2px underline indicator. Optional .tab-count badges sit after the label.

The parts

One underline tab set with a count badge and a disabled tab.

Underline
  1. Container .tabs — role="tablist" with an aria-label. Owns the bottom hairline that the active underline sits on.
  2. Tab .tab — a real <button role="tab">. Selected state via aria-selected="true" (preferred) or .is-active.
  3. Indicator ::after — the 2px brand underline, inset from the label edges; drawn only on the active tab.
  4. Count .tab-count — mono badge showing inventory; recolours to brand on the active tab.
  5. Disabled [disabled] — dims via the disabled foreground token; never carries a count.
  6. Panel — the content region the tabs switch (your markup); associate with role="tabpanel" + aria-labelledby.

02Variants

Two variants share one markup — the only difference is .tabs-pill on the container. Underline is the default for view-level navigation; pill is for compact, in-component switching where an underline would compete with surrounding chrome.

Underline .tabs

Page- or panel-level navigation. Sits flush on a surface edge; the active underline ties the selected tab to the content below.

03Pill / segmented

The .tabs-pill control is a self-contained segmented switch: a rounded shell with a subtle fill, where the active segment lifts onto a white surface with a soft shadow — no underline. Use it for short, mutually-exclusive choices that act in place (a filter or view toggle) rather than navigating to a different panel.

Default .tabs-pill

Segments size to their content. Good when labels differ in length.

Auto width

Equal segments .tabs-pill.is-equal

Every segment the same width regardless of label. The classic symmetric control for 2–4 peer options.

Equal width

Full width .tabs-pill.is-block

Stretches to fill its container with equal segments — for narrow columns, drawers and mobile sheets.

Block

Icon-only .tab.is-icon

Square segments for view toggles where the icon is self-explanatory. Always supply an aria-label — there's no visible text.

05Counts & icons

Tabs can carry a leading icon, a trailing count, or both. Counts show inventory so users choose a tab knowing what's behind it; icons aid scanning in pill toolbars. Don't use both a count and an icon on the same tab — it crowds the label.

Count badges .tab-count

Right of the label. Inactive badges are neutral grey; the active tab's badge picks up the brand tint to stay legible against the bold label.

Leading icons

Icons sit before the label at 15px to match the type. Best in pill toolbars where labels are short.

08Mobile

Tabs are dense, horizontal and tap-driven — three things that fight a narrow screen. Below is the same component on a phone, plus the rules that keep it usable. Resize this page narrow (or rotate a device) and the sliding indicator realigns automatically.

On a phone

Underline tabs that overflow use .is-scroll (swipe sideways); the filter uses a full-width pill .is-block. Both animate and meet the 44px touch target.

Live

What changes on mobile

Nothing about the markup — just three opt-in classes and the platform touch minimum kicking in:

  • Overflowing underline tabs get .is-scroll → swipe, no page-wide horizontal scroll.
  • Filters become .tabs-pill.is-block → full-width, equal segments, thumb-friendly.
  • On a coarse pointer every tab is bumped to a ≥44px hit height automatically.
  • The sliding indicator recomputes on resize / rotation, so it never drifts.

09When to use

Tabs switch between peer views of one subject without leaving the page. If the choice is sequential, hierarchical, or changes the page, reach for a different pattern.

SituationUseWhy
Peer views of one record (Overview / Transactions / Settings)Tabs · underlineNon-sequential, same context, no page change.
Compact filter inside a card or toolbar (Day / Week / Month)Tabs · pillSegmented control reads as an in-place filter.
A step-by-step flow (verify → fund → sign)StepperOrder matters and steps gate each other — tabs imply free movement.
Position in a hierarchy (Org › Project › Task)BreadcrumbsTabs are flat; breadcrumbs show depth.
Picking one value to submit in a formRadio / segmented inputTabs navigate; they aren't a form field.
More than ~6 sections, or labels that wrapSide nav or a selectTabs crowd and lose their horizontal scan past ~6.
Independent sections all potentially open at onceAccordionTabs show one panel at a time by definition.

10Don'ts

Don't tone individual tabs. The tab chrome is neutral; put colour in the count badge, not the tab body.

Don't let tabs wrap to two rows. Past ~6 tabs the row breaks and the indicator gets confusing. Use a side nav or select.

Don't use tabs for a sequence. Numbered steps that gate each other are a Stepper, not tabs.

Don't mark two tabs active. Exactly one aria-selected="true" at a time — tabs show a single panel.

11Accessibility

Tabs follow the WAI-ARIA tabs pattern: a tablist of tab buttons controlling tabpanel regions, with roving tabindex so the arrow keys (not Tab) move between tabs.

Keyboard contract

KeyAction
TabMove focus into the tab list (lands on the active tab), then out to the panel.
← / →Move focus to previous / next tab. Wraps at the ends.
Home / EndJump to the first / last tab.
Enter / SpaceActivate the focused tab (if not auto-activating on focus).
