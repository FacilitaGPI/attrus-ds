Design system

TypeScript layer

Every component ships as a typed React component (.tsx) wrapped around the canonical CSS. The CSS stays the visual source of truth; TypeScript adds the contract — and the tooling that keeps both in sync.

01Architecture — two layers, one truth

The .tsx files carry no styling. They compose the canonical classes — .btn, .drawer, .pill — and expose them as typed props. Change a token and every component reskins; the TypeScript never needs to know.

The division of labour

CSS guarantees everything looks right. TypeScript guarantees everything is used right.

File anatomy components/<Name>/

Each component is a folder with three files. The visual rules live elsewhere — in preview/components/*.css.

Per component
FileRole
<Name>.tsxThe component. Maps typed props → canonical classes. Zero hardcoded styles; behaviour only (open/close, keyboard, ARIA).
<Name>.d.tsThe public contract, mirrored 1:1 — union types for every variant, doc comment on every prop. What consumers and tooling read.
<Name>.htmlThe bench — a live preview that demonstrates every variant, prop and export, with interaction tests. The card you see in the DS tab.

Example — variant as a union type

The type IS the documentation. An invalid value fails the build with the exact line.

Button.d.ts
export type ButtonVariant = 'primary' | 'secondary' | 'tertiary' | 'ghost' | 'danger' | 'danger-outline' | 'success' | 'link' | 'inverse'; // Each value maps 1:1 to a canonical class in buttons.css — // the drift linter proves it on every run.

02What the types add

Before the migration, components were classes assembled by hand — and a typo was an unstyled element nobody caught. The types turn those silent failures into build errors, and the API into data that tooling can audit.

Explicit, autocompletable API

Every variant, size and slot is a union type. Consumers get autocomplete and inline docs — no opening the CSS to discover what exists.

Errors at build time

variant="primry" breaks the build with a line number. Misspelt classes used to ship as unstyled UI.

Safe refactors

Rename or remove a prop and the compiler flags every call site at once — no grep-and-pray.

Machine-auditable source of truth

Union types are structured data. The drift linter cross-checks CSS ↔ types ↔ benches automatically.

Zero-translation handoff

The Next.js app in export/nextjs/ imports these same .tsx files directly — design system and product consume one artifact.

Styling stays protected

Types add a contract on top of the CSS — they never replace it. Tokens, themes and dark mode keep working untouched.

03Consuming the typed components

Two consumption modes, same source files.

In the browser _ds_bundle.js

The compiler bundles every .tsx and exposes them on a window namespace. This is how the benches and demo pages work — React UMD + one script tag.

No build step
<!-- after React UMD --> <script src="_ds_bundle.js"></script> const { Button, Drawer, DataTable } = window.<Namespace>; React.createElement(Button, { variant: 'primary', size: 'sm' }, 'Save');

In a real app import

TypeScript projects import the source directly — full type-checking, tree-shaking, no bundle. The reference implementation is export/nextjs/.

Next.js / Vite
// app code — same files the design system compiles import { Button } from '…/components/Button/Button'; import '…/colors_and_type.css'; // tokens + themes import '…/preview/components/buttons.css'; // canonical visual <Button variant="secondary" size="sm" loading>Export</Button>
The CSS imports are mandatory — the components only compose class names. Without the canonical stylesheets they render unstyled (by design: one visual source of truth).

Three artifacts watch the contract so humans don't have to.

05Adding or changing a component

The order matters: visual first, contract second, proof third.

  1. CSS canônico primeiro
    Create or extend the classes in preview/components/*.css, consuming only semantic tokens. The visual decision lives here.
  2. .tsx compõe — nunca estiliza
    Map props → classes. Behaviour (keyboard, ARIA, open/close) is welcome; hardcoded colours and sizes are not.
  3. Espelhe o contrato no .d.ts
    Union type for every variant, doc comment on every prop. This is the file consumers and tooling read.
  4. Bench com cobertura completa
    Demonstrate every variant, prop and export in <Name>.html — with a live interaction test for anything stateful.
  5. Linter verde, duas vezes
    Run tools/drift-linter.html: CSS ↔ types must read zero drift; bench coverage must read cobertura completa.
